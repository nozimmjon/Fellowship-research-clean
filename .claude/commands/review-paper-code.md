# Review Paper Code

Review a research project's paper and code for reproducibility, code quality, and paper-code alignment. Be constructive, concrete, and calibrated. Treat gaps as items to verify, not accusations.

## Scope

This skill supports:
- Quarto manuscripts (`.qmd`) with R inline expressions
- R code files (`.R`, `.r`)
- CSV-based output tables in `outputs/tables/`
- Markdown includes in `reports/includes/`

Default review depth:
- `main`: prioritize the main manuscript, main R scripts, and core outputs
- `full`: inspect all detected code files in scope

If no depth is provided, default to `main`.

## Phase 1: Discover the Project

First parse `$ARGUMENTS`:
- If one argument is `main` or `full`, use it as `REVIEW_DEPTH`.

### 1. Find the manuscript

Use Glob to search for `**/*.qmd`, focusing on the `reports/` directory.

Identify the main manuscript as the `.qmd` file that contains the most `{{< include ... >}}` directives or inline R expressions, or is named `00_main.qmd`.

Also find:
- All `{{< include ... >}}` referenced files (typically in `reports/includes/`)
- Supporting `.qmd` files: policy brief, slides, technical appendix
- `_quarto.yml` for project configuration

Record the result as `PAPER_FILES` (list of all manuscript-related files).

### 2. Find the code

Look for R code in this order:
- `R/` directory (primary)
- `scripts/` directory
- `analysis/` directory
- Root-level `.R` files (`run_pipeline.R`, `_targets.R`, etc.)

Record the result as `CODE_DIR` and `CODE_FILES_ALL`.

### 3. Find code files

Within `CODE_DIR` and subdirectories, find:
- `**/*.R`
- `**/*.r`

If `REVIEW_DEPTH = main`, prioritize:
- Master scripts such as `run_pipeline.R`, `_targets.R`, `main.R`
- Files referenced by those scripts (via `source()`)
- Files that generate tables/figures (look for `write_csv`, `ggsave`, `saveRDS`)
- Cap the initial review set at a reasonable number

If `REVIEW_DEPTH = full`, include all detected code files.

Record:
- `CODE_FILES_ALL`
- `CODE_FILES_REVIEWED`

### 4. Find outputs and supporting documentation

Look for:
- `outputs/tables/*.csv` — the CSV tables consumed by the manuscript
- `outputs/figures/` — generated figures
- `outputs/models/` — saved model objects
- `README.md`, `REPO_STATUS.md`, `AGENTS.md`
- `research_strategy.md` or similar design documents
- `renv.lock`, `DESCRIPTION`
- `references.bib`

Record relevant files as available.

### 5. Handle ambiguity gracefully

If you find a manuscript and at least some code, continue even if discovery is imperfect.

Only stop if you cannot find either:
- a main `.qmd` manuscript file, or
- any relevant R code files

Before proceeding, tell the user:
- the manuscript file(s) chosen
- the code directory chosen
- the number of code files detected and the number selected for review
- the review depth
- any ambiguity worth noting

## Phase 2: Read the Manuscript

Read all `PAPER_FILES` (main `.qmd` and all includes).

Extract a compact working summary for later cross-checking:
- Paper title
- Main research question
- Main sample description (waves, countries, age groups)
- Main data sources
- Main dependent variables
- Main explanatory variables
- Main estimation methods
- Fixed effects and clustering, if stated
- Main sample restrictions
- All inline R expressions that reference CSV files (e.g., `read_csv_safe(...)`)
- All accessor function calls (e.g., `fmt_num(...)`, `fmt_pct(...)`)
- Headline quantitative claims

Store this as `PAPER_SUMMARY`.

## Phase 3: Launch 2 Agents in Parallel

In a single message, launch both agents using the Agent tool with `subagent_type: "general-purpose"`.

---

### AGENT A: Code Reproducibility and Quality

Store as `CODE_REVIEW_SUMMARY`.

Prompt:

> You are reviewing research code for reproducibility and code quality in a Quarto + R social science project.
>
> Files in scope:
> - Reviewed code files: [insert `CODE_FILES_REVIEWED`]
> - README / documentation files: [insert discovered supporting files or "none found"]
> - Output CSVs: [insert list of CSV files in outputs/tables/]
>
> Review the files and produce a compact report focused on the most decision-relevant findings.
>
> Check:
> 1. Hardcoded absolute paths or machine-specific assumptions
> 2. Randomized procedures without an obvious seed
> 3. Outputs (CSVs, figures) that the manuscript consumes but are not generated by the reviewed pipeline
> 4. Data inputs and whether path conventions are consistent (look for `file.path()`, `here()`, or relative paths)
> 5. Dependency management (renv.lock, library calls)
> 6. Run order: is there a master script (`run_pipeline.R`, `_targets.R`) that documents execution order?
> 7. Large commented-out blocks, weak script structure, or hard-to-follow long files
> 8. Opaque transformations: unexplained filters, recodes, merges, or thresholds important for interpretation
> 9. **Hardcoded numbers in .qmd files**: Check ALL `.qmd` files for numbers that should be pulled from CSVs via R inline expressions but are instead written as plain text. This is a CRITICAL reproducibility issue.
> 10. **CSV accessor pattern**: Verify that `read_csv_safe()` calls in the manuscript match actual CSV files that exist in `outputs/tables/`
>
> Use these labels:
> - PASS: looks solid
> - NOTE: minor improvement opportunity
> - VERIFY: worth human confirmation
> - MISSING: expected file or documentation is absent
> - HARDCODED: number in manuscript not pulled from pipeline
>
> Output exactly these sections:
>
> ## Overall
> 3-6 bullets on the overall state of the codebase.
>
> ## Top Findings
> Up to 12 items total, ordered by importance.
> Format: - [LABEL] Short finding title — file(s): line reference(s) — why it matters — what to check next
>
> ## Hardcoded Numbers Audit
> List every plain-text number in .qmd files that should be an R inline expression.
> Format: - [HARDCODED] file:line — "quoted text" — which CSV should supply this value
>
> ## Strengths
> 3-8 bullets with genuine positives.
>
> ## Reproducibility Checklist
> One line each:
> - Relative paths: PASS / NOTE / VERIFY / MISSING
> - Random seed practice: PASS / NOTE / VERIFY / MISSING
> - Outputs generated by pipeline: PASS / NOTE / VERIFY / MISSING
> - Dependency management: PASS / NOTE / VERIFY / MISSING
> - Run order: PASS / NOTE / VERIFY / MISSING
> - README / documentation: PASS / NOTE / VERIFY / MISSING
> - Manuscript pulls from CSVs (no hardcoding): PASS / NOTE / VERIFY / MISSING
>
> ## File Notes
> Brief notes only for files with VERIFY, NOTE, HARDCODED, or especially strong positive signal.

### AGENT B: Paper-to-Code Mapping

Store as `MAPPING_SUMMARY`.

Prompt:

> You are mapping a Quarto research manuscript's empirical claims to its R code implementation.
>
> Inputs:
> - Paper summary: [insert `PAPER_SUMMARY`]
> - Reviewed code files: [insert `CODE_FILES_REVIEWED`]
> - Code directory: [insert `CODE_DIR`]
> - Output CSVs: [insert list of CSV files in outputs/tables/]
> - Research strategy document: [insert path if found, or "none found"]
>
> Read the code files and CSV outputs as needed. Map the paper's core empirical design to the code.
>
> Focus on:
> 1. Main tables and figures — do the CSVs consumed by the manuscript get produced by the R pipeline?
> 2. Main variables — are variable constructions in code consistent with manuscript descriptions?
> 3. Sample restrictions — does code apply the same filters described in the paper?
> 4. Estimation methods — do R model specifications match what the paper describes?
> 5. Fixed effects and clustering — do `feols()` or equivalent calls match stated specifications?
> 6. **Research strategy alignment** — if a `research_strategy.md` or similar document exists, check whether the actual code matches pre-registered equations and specifications. Flag deviations (e.g., dropped controls, changed functional form).
> 7. **Inline expression tracing** — for key inline R expressions in the manuscript, trace them back to the CSV that supplies them and the R script that generates that CSV.
>
> Use these confidence labels:
> - HIGH: clear and specific match
> - MEDIUM: plausible match but not airtight
> - LOW: weak or indirect match
> - NOT FOUND: no plausible match
> - DEVIATION: code differs from pre-registered specification
>
> Output exactly these sections:
>
> ## Verified Matches
> Up to 12 bullets: Paper element -> Code evidence -> HIGH / MEDIUM -> brief note
>
> ## Pre-Registration Deviations
> Up to 8 bullets: Specification element -> What research_strategy says -> What code does -> DEVIATION -> impact assessment
>
> ## Items To Verify
> Up to 12 bullets: Paper element -> Code evidence or absence -> LOW / NOT FOUND / MEDIUM -> why this deserves a check
>
> ## Likely Discrepancies
> Up to 8 bullets where paper and code point in different directions.
>
> ## Coverage Notes
> 3-6 bullets on what was easy to match, what was ambiguous, and what may sit outside reviewed files.

## Phase 4: Synthesize

After both agents return, synthesize the results yourself.

Compare the two outputs for agreement and tension. Downgrade overconfident claims. Note where limited file coverage weakens confidence.

Create:
- `OVERALL_ASSESSMENT`: 2-4 sentences leading with what works
- `TOP_ACTIONS`: 3-8 concrete next steps, ordered by importance
- `MATCHED_ITEMS`: high-confidence paper-code matches
- `VERIFY_ITEMS`: gaps or ambiguous matches worth checking
- `NOT_FOUND_ITEMS`: important paper elements with no plausible code match
- `DEVIATIONS`: pre-registration deviations
- `HARDCODED_ITEMS`: all hardcoded numbers found in .qmd files
